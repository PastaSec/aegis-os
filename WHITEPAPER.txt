======================================================================
ABSTRACT
======================================================================

AEGIS OS is an offline-first community resilience platform designed to preserve knowledge, maintain local communication, and support small-community coordination when normal infrastructure is unreliable or unavailable.

The first reference implementation is a Raspberry Pi 3 Model B cyberdeck equipped with an Adafruit 2.8 inch PiTFT resistive touchscreen. This device, called the AEGIS Console, is not the full system. It is the operator interface for a broader platform that may include a local web portal, offline document library, community bulletin board, portable receiver nodes, ESP32 sensor nodes, OpenWrt routers, and dockable USB instruments such as the Flipper Zero.

The project is built around three core principles:

Knowledge. Communication. Resilience.

AEGIS OS is intended for practical local use: a household, a block, or a small neighborhood. Its purpose is not to replace emergency services, public safety infrastructure, commercial networking, or the Internet. Its purpose is to provide a local fallback when those systems are delayed, degraded, overloaded, or absent.

If everything goes sideways, Ian and his neighbors should still have a place to connect, communicate, learn, coordinate, and share resources.

======================================================================
1. PROJECT IDENTITY
======================================================================

1.1 Name

AEGIS OS stands for Autonomous Emergency & General Information System.

The word "aegis" historically refers to a shield or protective covering. This project adopts that meaning deliberately. AEGIS is not a weapon. It is not a fear-driven project. It is a protective system based on knowledge, communication, preparation, and community coordination.

1.2 Repository

The official repository name is:

PastaSec/aegis-os

The name "aegis-os" was chosen because the project is broader than a single Raspberry Pi build. The Raspberry Pi cyberdeck is the first hardware target. The larger project is a portable operating platform that can evolve across devices.

1.3 Motto

Knowledge. Communication. Resilience.

These are the three filters used to evaluate every proposed feature.

A feature belongs in AEGIS if it improves at least one of the following:

- access to useful knowledge
- communication between people
- coordination of community resources
- operational awareness
- resilience during degraded infrastructure

A feature does not belong if it exists only because it is interesting, flashy, or technically possible.

======================================================================
2. MISSION
======================================================================

AEGIS OS exists to help a small community remain informed, connected, and capable when normal infrastructure becomes unreliable.

The mission is:

Build a self-contained community operations platform capable of monitoring, communicating, documenting, educating, and coordinating a small community without relying on external infrastructure.

This means AEGIS must be useful when:

- Internet connectivity is unavailable
- cellular networks are overloaded or down
- power is intermittent
- local information is scattered
- people need to coordinate help, supplies, or status updates
- trusted knowledge must be available offline
- normal communication channels are unreliable

AEGIS is not designed around a single disaster scenario. It is designed around general degradation: hurricanes, power outages, network outages, local emergencies, evacuations, supply disruptions, or any situation where neighbors need a local coordination point.

======================================================================
3. DESIGN PHILOSOPHY
======================================================================

3.1 Offline First

AEGIS must not depend on the cloud for core operation.

Cloud services may enhance the system when available, but the central functions must remain local:

- offline documents
- local search
- local web portal
- local notes
- local logs
- local authentication
- local service status
- local message board

Internet access is treated as a bonus, not a requirement.

3.2 Community First

AEGIS is not only an operator tool. It is a community tool.

The operator console is important, but the larger value is the ability for neighbors to connect to a local portal using ordinary phones, tablets, and laptops.

The community portal should be simple enough for non-technical users.

A neighbor should not need to know Linux, SSH, ham radio, mesh networking, or Raspberry Pi administration. They should be able to connect to a local Wi-Fi network and open a browser.

3.3 Modular by Design

AEGIS should not become a fragile pile of scripts.

Every major capability should become a module:

- dashboard
- documents
- search
- network
- portal
- monitoring
- sensors
- Flipper
- mesh
- notes
- AI
- system tools

Modules should expose clear capabilities, health status, configuration, and logs.

3.4 Observable Systems

Everything important should be observable.

The operator should be able to see:

- Pi temperature
- undervoltage / throttling state
- RAM usage
- disk usage
- network state
- Wi-Fi clients
- local services
- portal health
- connected nodes
- USB expansion devices
- sensors
- batteries and power sources when supported

If AEGIS becomes infrastructure, it must be monitored like infrastructure.

3.5 Graceful Degradation

AEGIS should remain useful even when some modules fail.

If the community portal fails, the console should still work.
If the mesh layer fails, the offline library should still work.
If the touchscreen fails, SSH should still work.
If Wi-Fi is down, local console tools should still work.
If Internet access is gone, local docs and logs should still work.

The system should never depend on a single fragile path.

======================================================================
4. CURRENT REFERENCE HARDWARE
======================================================================

The current AEGIS Console reference hardware is:

- Raspberry Pi 3 Model B Rev 1.2
- Debian GNU/Linux 13 Trixie
- 64-bit aarch64 userspace
- Adafruit 2.8 inch PiTFT resistive touchscreen
- ILI9341 / fb_ili9340 framebuffer display
- STMPE610 resistive touch controller
- SPI enabled
- `/dev/fb1` mapped to the PiTFT
- Wi-Fi enabled
- Bluetooth available
- SSH enabled

The PiTFT is configured with the following overlay:

    dtoverlay=pitft28-resistive,rotate=90,speed=32000000,fps=20

The display framebuffer appears as:

    /dev/fb1

The HDMI / primary framebuffer appears as:

    /dev/fb0

A systemd service maps tty1 to the PiTFT framebuffer after boot.

This hardware is not the final definition of AEGIS. It is simply the first working hardware target.

======================================================================
5. SYSTEM MODEL
======================================================================

AEGIS OS is best understood as three connected product layers.

----------------------------------------------------------------------
5.1 AEGIS Console
----------------------------------------------------------------------

The AEGIS Console is the handheld or portable operator device.

In the first build, this is the Raspberry Pi 3 with PiTFT and keyboard.

Primary responsibilities:

- boot into a readable dashboard
- display system health
- launch operator tools
- access the offline library
- manage the local portal
- monitor nodes
- provide SSH access to trusted systems
- detect expansion devices
- operate without a desktop environment

The console is not meant to be a tiny laptop. The 320x240 display is best suited to purpose-built terminal interfaces, menus, dashboards, and status screens.

----------------------------------------------------------------------
5.2 AEGIS Hub
----------------------------------------------------------------------

The AEGIS Hub provides local services.

The hub may run on the same Raspberry Pi as the console, or it may eventually run on a stronger device such as a home server, mini PC, or dedicated Raspberry Pi.

Primary responsibilities:

- local web portal
- offline document hosting
- local search index
- announcements
- message board
- resource inventory
- file sharing
- service health API
- authentication
- logs and backups

The hub is what neighbors connect to.

----------------------------------------------------------------------
5.3 AEGIS Nodes
----------------------------------------------------------------------

AEGIS Nodes are small devices that extend the system.

Possible node types:

- Raspberry Pi Zero W / Zero 2 W receiver nodes
- ESP32 sensor nodes
- OpenWrt routers
- USB serial devices
- battery monitor nodes
- future LoRa nodes
- future GPS/weather nodes

Nodes should be low cost, reproducible, and replaceable.

======================================================================
6. COMMUNITY PORTAL
======================================================================

The community portal is one of the most important parts of AEGIS.

The portal is the user-facing system for neighbors. It should work through an ordinary web browser and should not require installing an app.

A first version of the portal should include:

- welcome page
- current announcements
- local status updates
- offline document links
- message board
- request help / offer help board
- resource inventory
- local map or neighborhood notes
- emergency contacts
- radio/channel plan if applicable
- system status page

The portal should be designed for clarity rather than style.

During a stressful situation, users need large buttons, plain language, and obvious actions.

Example top-level portal sections:

    Announcements
    Need Help
    Offer Help
    Resources
    Medical
    Offline Library
    Neighborhood Map
    Contact Board
    System Status

The portal should support roles:

- Admin: full control
- Operator: manage posts, docs, and status
- Resident: post messages and requests
- Guest: read public information

Early versions may use simple authentication. Later versions should include better user management.

======================================================================
7. OFFLINE KNOWLEDGE BASE
======================================================================

The offline knowledge base is a core AEGIS function.

Information is infrastructure.

When connectivity is down, people still need access to reliable documents. AEGIS should store and serve information locally.

Initial document categories:

- Wikipedia / Kiwix ZIM files
- first aid
- CPR
- wilderness medicine
- hurricane preparedness
- water purification
- sanitation
- food safety
- shelter
- power and generators
- communications
- networking
- Linux administration
- Raspberry Pi documentation
- OpenWrt documentation
- ESP32 documentation
- local SOPs
- family documents where appropriate and encrypted
- community procedures

The long-term goal is one search interface that can search:

- ZIM files
- PDFs
- Markdown
- plain text
- HTML
- notes
- logs
- manuals

The search system should be local and should not require the Internet.

The 320x240 console may use a terminal-first viewer. The community portal may provide a larger browser-based document interface for phones and laptops.

======================================================================
8. MONITORING
======================================================================

AEGIS should monitor everything available in the current technology stack.

Monitoring begins with the Pi itself:

- CPU temperature
- CPU load
- RAM usage
- disk usage
- undervoltage / throttling state
- uptime
- network interfaces
- Wi-Fi signal
- connected USB devices
- running services
- logs

The next monitoring layer is AEGIS services:

- portal service
- document service
- search service
- database
- DNS/DHCP if used
- access point status
- backup status

The next layer is connected nodes:

- node online/offline state
- battery level if available
- sensor readings
- last heartbeat
- firmware version
- location/role label

Possible environmental readings:

- temperature
- humidity
- light
- motion
- noise
- air quality if hardware is added
- water level if hardware is added
- battery / solar if hardware is added

Monitoring is not for decoration. It is how the operator knows whether the system can be trusted.

======================================================================
9. COMMUNICATION AND NETWORKING
======================================================================

AEGIS should support multiple communication paths.

Initial communication methods:

- local Wi-Fi access point
- existing home Wi-Fi client mode
- Ethernet where available
- SSH
- local web portal

Future communication methods:

- OpenWrt mesh experiments
- Raspberry Pi node relays
- ESP32 low-power status nodes
- LoRa where legal and appropriate
- amateur radio integration where licensed and appropriate
- USB serial
- Bluetooth experiments

The goal is not to build an illegal or disruptive radio system.

The goal is lawful local connectivity and practical fallback communication.

A realistic first community network might consist of:

- AEGIS Console as admin terminal
- AEGIS Hub as portal/document server
- one or more OpenWrt routers for wider local Wi-Fi coverage
- Pi Zero nodes for receiver/display endpoints
- ESP32 nodes for simple sensor/status reporting

The network should be simple enough to explain and deploy.

======================================================================
10. FLIPPER ZERO AS EXPANSION MODULE
======================================================================

The Flipper Zero should not become the heart of AEGIS.

It should be treated as a dockable instrument.

This mirrors real field practice: a multimeter, radio, GPS, or analyzer is not the command center. It is a tool plugged in when needed.

When connected over USB, AEGIS should eventually detect the Flipper and expose supported workflows:

- identify device
- show connection status
- open serial console if applicable
- access files where supported
- log interactions
- launch approved utilities
- document workflows

All Flipper-related features should remain lawful and user-controlled. AEGIS should not automate unauthorized access, interference, credential theft, or disruptive actions.

The guiding idea is:

    The Pi is the brain.
    The Flipper is an instrument.

======================================================================
11. ESP32 AND SENSOR NODES
======================================================================

ESP32 devices are valuable because they are low cost, low power, and easy to deploy.

Possible ESP32 node roles:

- temperature / humidity sensor
- light sensor
- motion sensor
- sound/noise level sensor
- simple status beacon
- battery monitor
- door/window status
- water level sensor
- remote button / alert device
- local display receiver

The XIAO ESP32S3 Sense is especially interesting as a future sensor/camera/audio module.

AEGIS should treat ESP32s as modular nodes with:

- node ID
- role
- firmware version
- last seen timestamp
- battery state if available
- sensor payload
- health status

======================================================================

AEGIS should be organized as a modular Python-first project with room for additional services.

Proposed structure:

    aegis-os/
        README.md
        WHITEPAPER.txt
        install.sh
        pyproject.toml
        aegis/
            core/
                config.py
                events.py
                logging.py
                plugins.py
                state.py
            ui/
                dashboard.py
                menu.py
                widgets/
            plugins/
                monitoring/
                docs/
                network/
                portal/
                flipper/
                esp32/
                notes/
                ai/
            portal/
                app.py
                templates/
                static/
        docs/
        assets/
        systemd/
        scripts/
        data/

The core should provide:

- configuration loading
- event bus
- plugin discovery
- logging
- state store
- service health registry
- command execution wrappers

Each plugin should expose:

- name
- description
- version
- status
- menu entries
- health check
- optional background service
- optional web routes

The UI should not hard-code every feature. It should discover available modules.

======================================================================
13. DATA MODEL
======================================================================

Early AEGIS should use simple local storage.

Recommended data layers:

- plain files for documents
- SQLite for portal data
- JSON or YAML for configuration
- rotating text logs for system events
- Markdown for operator notes

Important data types:

- announcements
- messages
- help requests
- offers
- resources
- nodes
- sensor readings
- incidents
- documents
- users
- service health records

Data should be easy to back up and recover.

Complex distributed databases are not appropriate for early versions.

======================================================================
14. SECURITY MODEL
======================================================================

AEGIS is a local community system, but security still matters.

Initial security goals:

- preserve SSH access for the operator
- avoid exposing admin controls publicly
- separate admin/operator/resident/guest capabilities
- log administrative actions
- avoid storing sensitive family documents unencrypted
- avoid default passwords
- avoid unnecessary services
- keep recovery paths available

The system should assume that the local network may include unknown devices.

Public information can be public. Operational controls should not be.

======================================================================
15. LOGO AND VISUAL IDENTITY
======================================================================

The AEGIS insignia should represent Florida, community, knowledge, and communication.

The logo should avoid false geography. Mountains do not fit the project identity because the reference community is in Florida.

Preferred symbolism:

- central A for AEGIS
- shield-like structure for protection
- open book for preserved knowledge
- signal arcs for communication
- water or horizon line for Florida/coastal identity
- community symbol for neighbors
- clean circular or shield framing
- high-contrast monochrome for engraving and boot screens

The logo should work as:

- application icon
- PiTFT boot mark
- GitHub repository mark
- sticker
- laser engraving
- field manual cover
- equipment label

The identity should feel practical, local, and trustworthy.

It should not feel like generic AI art, corporate software branding, or fear-based survival marketing.

======================================================================
16. MVP DEFINITION
======================================================================

The first complete AEGIS MVP is achieved when the system can do the following:

1. Power on and boot reliably.
2. Display an operator dashboard on the PiTFT.
3. Maintain SSH fallback access.
4. Show system health, including power warnings.
5. Provide keyboard-driven menus.
6. Access an offline document library.
7. Search local text/Markdown/PDF documents.
8. Host a local web portal over Wi-Fi.
9. Allow a phone to connect and view local resources.
10. Provide a simple message or bulletin board.
11. Provide a safe shutdown workflow.
12. Preserve logs and notes locally.

This MVP does not require mesh networking, LoRa, AI, Flipper integration, or sensor nodes.

Those are expansion phases.

======================================================================
18. NON-GOALS
======================================================================

AEGIS OS should not become:

- a laptop replacement
- a desktop environment project
- a media center
- a gaming handheld
- an offensive hacking platform
- a cloud-only app
- a brittle pile of undocumented scripts
- a system that requires expert knowledge for neighbors to use

AEGIS may include advanced tools for the operator, but the mission remains community resilience.

======================================================================
19. SUCCESS CRITERIA
======================================================================

AEGIS succeeds if:

- Ian can carry it, power it, and operate it without a monitor.
- It provides useful local information when offline.
- A neighbor can connect with a phone and understand what to do.
- The system can host local announcements and resource information.
- The operator can monitor system health.
- The system can be rebuilt from documentation.
- Expansion modules can be added without rewriting everything.
- The project remains useful even when some features are unavailable.

AEGIS fails if:

- it becomes too complicated to operate under stress
- it depends on cloud services for core functions
- it becomes a collection of unrelated scripts
- it requires a desktop monitor to recover
- it ignores the needs of non-technical neighbors
